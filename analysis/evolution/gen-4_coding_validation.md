## 2. 문제 코드 심층 분석

### 2.1 LEX-HEDGE (F1=0.732) — Recall 부족

**문제**: regex 패턴이 지나치게 구체적인 완성 문장만 탐지. 단순하면서 가장 흔한 헤지 표현을 놓침.

**놓치는 표현 (500건 중 빈도)**:
| 미탐지 표현 | regex 음성인데 포함된 건수 |
|------------|:-:|
| "~ㄹ 수 있" (일반형) | 329건 |
| "것 같" | 19건 |
| "~ㄹ 수도 있" | 14건 |
| "모르겠" | 11건 |
| "~일지도" | 7건 |

**원인**: 현재 패턴 `일\s*수\s*(?:도\s*)?있(?:습니다|어요|다)`는 종결어미를 요구하여, "~할 수 있지만", "~될 수 있는" 같은 연결형을 놓침. "것 같다" 패턴도 `~` 또는 `\.\.\.` 선행을 요구하여 일반적 사용을 놓침.

**개선안**:
```python
# 기존
r"일\s*수\s*(?:도\s*)?있(?:습니다|어요|다)",
r"(?:~|\.\.\.)\s*(?:것\s*같습니다|것\s*같아요|것\s*같다)",

# 개선
r"[ㄹ을를]\s*수\s*(?:도\s*)?있",                    # 종결어미 제한 제거
r"것\s*같(?:습니다|아요|다|은데|지만|고)",           # 선행 조건 제거 + 연결어미 추가
r"모르겠(?:습니다|어요|다|지만|는데)",               # 새 패턴 추가
```

### 2.2 LEX-FORMAL (F1=0.757) — Recall 부족

**문제**: 기본 격식체 종결어미(`-습니다`, `-입니다`)를 가진 글이 regex 특수 표현(`살펴보겠습니다`, `분석하면` 등)에 매칭되지 않아 FN 발생.

**원인**: 패턴이 "학술적 격식체"에 집중되어, 일반적인 격식체 문장(`~하고 있습니다`, `~한다고 합니다`)을 놓침.

**개선안**:
```python
# 추가 패턴
r"(?:하|되|인|한|는|일|을)\s*것입니다",
r"(?:됩니다|입니다|습니다)\s*[。.]\s*$",   # 격식 종결 + 마침표
r"(?:하였|되었|이었)습니다",                # 과거형 격식체
```

### 2.3 PRAG-DISAGREE (F1=0.500) — Precision 부족

**문제**: `아닌데(?:요)?` 패턴이 실제 반대 의견이 아닌 문맥에서 오탐지. 예: "~가 아닌데도 불구하고", "핑계 아닌데" 등 서술적 용법.

**패턴별 히트 분석**:
- `아닌데(?:요)?`: 8건 (대부분 FP — 서술적 용법)
- `반대(?:합니다|해요|로\s+생각)`: 1건
- `오히려\s+(?:반대로|그\s+반대가)`: 1건

**원인**: "아닌데"는 대화에서 반박으로 쓰이기도 하지만, 게시글에서는 설명적 문맥에서 더 많이 등장. 게시글과 댓글을 구분하지 않고 동일 패턴을 적용하는 구조적 문제.

**개선안**:
```python
# 기존
r"아닌데(?:요)?",

# 개선: 문두 또는 독립절에서만 탐지
r"(?:^|\.\s+)아닌데(?:요)?(?:\s|$|,|\.)",   # 문장 시작일 때만
```

### 2.4 PRAG-AGREE (F1=0.714) — Precision 부족

**문제**: `맞습니다|맞아요|맞아(?:\s|$|,|\.)` 패턴이 인용문이나 서술 문맥에서 오탐지. 예: "~이 맞아요"가 사실 확인(fact-stating)인 경우.

**개선안**: 게시글에서는 PRAG-AGREE/DISAGREE 코드를 댓글 전용으로 제한하거나, 문맥 조건을 추가.

### 2.5 LEX-AI (F1=0.000) — 양성 사례 극소

**문제**: 500건 중 자동 탐지 양성이 1건뿐. AI 특유 표현 패턴이 지나치게 구체적이어서 대부분 놓침.

**관찰**: 실제 봇마당 에이전트들은 "AI스러운" 표현을 단순히 반복하지 않고 변형하여 사용. 현재 패턴은 한국어 AI 챗봇의 전형적 응답 패턴에 과적합.

**개선안**:
```python
# 추가 패턴 (더 넓은 포착)
r"안녕하세요[!.]\s*(?:오늘|이번)",            # AI 인사 + 도입
r"소개해\s*드리겠습니다",
r"정리해\s*드리겠습니다",
r"(?:핵심|주요)\s*(?:포인트|내용|사항)(?:을|는)",
r"마무리하겠습니다",
```

### 2.6 LEX-TECH (F1=0.788) — 도메인 범위 문제

**문제**: `가상자산|암호화폐|NFT|DeFi` 패턴이 145건으로 전체 LEX-TECH 히트의 최다수. 금융 콘텐츠(Taasfi 홍보 등)가 "기술 용어"로 분류되는 것이 의미론적으로 적절한지 의문.

또한 `모델` 패턴이 "사업 모델", "수익 모델" 같은 비기술적 용법도 탐지 (69건).

**개선안**:
```python
# 기존
r"가상자산|암호화폐|NFT|DeFi",
r"모델(?:\s+학습|\s+배포|\s+서빙)?",

# 개선: 금융 용어 분리 + 모델 문맥 제한
r"모델\s*(?:학습|배포|서빙|파라미터|가중치|추론|훈련)",  # 기술적 모델만
# 가상자산 관련은 별도 LEX-FINANCE 코드로 분리 권장
```

### 4.1 우선순위별 개선 항목

| 2 | PRAG-DISAGREE | Precision 0.333 — "아닌데" 과잉 탐지 | 중간 | 중간 |
| 3 | LEX-FORMAL | Recall 0.636 — 일반 격식체 누락 | 중간 | 낮음 |

### 4.2 구조적 개선 권고

1. **게시글/댓글 분리 코딩**: PRAG-AGREE, PRAG-DISAGREE는 본질적으로 대화적 행위(speech act). 게시글 본문에서는 FP가 높을 수밖에 없으므로, 댓글 전용 코드로 제한하거나 게시글용 임계값을 높이는 것을 권장.

2. **LEX-FINANCE 코드 신설**: `가상자산`, `암호화폐`, `NFT`, `DeFi`, `세금`, `절세`, `거래소` 등 금융 용어를 LEX-TECH에서 분리하여 별도 코드로 관리. Taasfi/BootingBot 홍보 콘텐츠가 "기술 담화"로 분류되는 오류 방지.

3. **LEX-HEDGE 패턴 확장**: 한국어 헤지 표현의 핵심인 `~ㄹ 수 있다` 계열이 전체 게시글의 65.8%(329/500)에서 사용되지만, 현재 regex는 특정 종결어미 조합만 잡음. 종결어미 제한을 제거하면 탐지율이 대폭 개선될 것.

4. **LEX-AI 패턴 재설계**: 현재 패턴은 사실상 무용(양성 1건). AI 에이전트의 "AI다움"을 탐지하려면, (a) 더 일반적인 AI 응답 패턴 추가, (b) TTR/반복구조 등 통계적 특성과 결합하는 복합 판정이 필요.

### 4.3 포화도 기여 추정

현재 코드 포화 차원(가중치 30%)이 0.73. 위 개선을 적용하면:
- LEX-HEDGE Recall 개선 → 은폐된 헤지 담화 패턴 노출 예상
- PRAG-AGREE/DISAGREE Precision 개선 → 삼각검증 정합성 향상

## 5. 방법론적 한계

1. **수동 검증의 재현성**: 수동 판정 기준도 키워드 기반(의미론적 확장)이므로 완전한 인간 코더 판정이 아님. 실제 Cohen's Kappa 기반 코더 간 신뢰도 검증과는 차이 존재.

2. **샘플 편향**: 양성/음성 균등 샘플링(15+15)은 코드별 유병률(base rate)을 반영하지 않음. 특히 양성이 극소(LEX-AI: 1건, PRAG-AGREE: 9건)인 코드는 표본 자체가 불충분.

3. **게시글 전용 검증**: 본 검증은 게시글(`pilot_posts.json`)만 대상. 댓글 데이터에서의 PRAG-AGREE/DISAGREE 성능은 게시글과 다를 수 있음(대화 맥락이 더 풍부하므로 Precision 향상 예상).
